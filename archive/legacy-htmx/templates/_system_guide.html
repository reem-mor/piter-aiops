Background · Architecture

How PITER AiOps works under the hood

When you run triage or ask the Live KB a question, this is the path from alert to grounded answer — no client-side RAG demo, real Amazon Bedrock RetrieveAndGenerate.

1 Browser HTMX POST /ask or /workflow/triage
→
2 Flask Validate question · route · render partial
→
3 Bedrock KB Vector retrieve chunks from S3 corpus
→
4 LLM Generate answer grounded in citations
→
5 UI Answer card · runbook · recommendation

What happens on each request

  1. Validation — Flask rejects empty, too-short, or stopword-only questions before any AWS call.
  2. RetrieveAndGenerate — boto3 calls Bedrock Agent Runtime with your KB ID and inference profile ARN.
  3. Citations — Retrieved chunk metadata is mapped to human-readable runbook filenames (not raw S3 URIs).
  4. Workflow reasoning — MVP triage combines live KB output with alert metadata (severity, MTTR baseline, escalation policy).
  5. Upload path — Optional S3 put + StartIngestionJob syncs new runbooks into the same KB prefix.

Deployment stack

Documents in S3 → Bedrock KB index → Flask container → Docker on EC2 → public URL for on-call.
